Subject

 

RESTful Web APIs

by Leonard Richardson  (Author),  Mike Amundsen (Author),  Sam Ruby  (Author)

"Syllabus"

  • April 1st: Chapters 1 & 2

    • Chapter 1, Surfing the Web This chapter explains basic terminology using a RESTful system you’re already familiar with: a website. 

    • Chapter 2, A Simple API This chapter translates the lessons of the Web to a programmable API with identical functionality to the website discussed in Chapter 1.

    • Discussion

      • What is a resource and what is a representation? A resource can be anything that has the property of unique addressability. In hypermedia, any access of an address "dereferences" your resource into a representation. This is what is unique about URLs.

      • What is the semantic challenge? The gap between how data is structured, and what that structure means. We often over-rely on the human component of our software (the user) to bridge this gap. APIs are meant to be consumed by computing agents other than humans. "Why bother to design APIs that serve self-descriptive messages if those messages won’t be understood by their software consumers?" 

      • Question for discussion: What are application semantics, and how can hypermedia types help you with them?

  • April 8th: Chapters 3 & 4

    • Chapter 3, Resources and Representations Resources are the fundamental concept underlying HTTP, and representations are the fundamental concept underlying REST. This chapter explains how they’re related. 

    • Chapter 4, Hypermedia Hypermedia is the missing ingredient that ties representations together into a co‐ herent API. This chapter shows what hypermedia is capable of, mostly using a hypermedia data format you’re already familiar with: HTML. 

    • Bonus material

      • H-Factors (subtly referenced on p54)

    • Discussion

      • What is the Web? URL, HTTP, HTML. Each has its own semantics. 

      • Are resources 1-to-1 with URLs? (p33)

      • Protocol Semantics of HTTP: what is the difference between Post-to-Append (p 37) and an Overloaded Post (p41)? Post-to-Append is a request to a resource to create a new resource. Overloaded post has no protocol semantics at all.

        • What is the difference between REST and SOAP?

      • What is hypermedia for? Guiding the request, promises about the response, workflow control. (p52)

        • Sidebar: What is transclusion?

      • How can we leverage the semantics given to us in standards to save us work when we are dealing with our application semantics?

      • How does the client know which requests it can make? (p45) Hypermedia.

      • What is hypermedia? "a way for the server to tell the client what HTTP requests the client might want to make in the future".

  • April 15th: Chapter 5 

    • Chapter 5, Domain-Specific Designs The obvious strategy is to design a completely new standard that deals with your exact problem. I use the Maze+XML standard as an example. 

    • Discussion

      • What is an API? Not from the book, but from Sedgewick's Algorithms: "The purpose of an API is to separate the client from the implementation: the client should know nothing about the implementation other than information given in the API, and the implementation should not take properties of any particular client into account. APIs enable us to separately develop code for various purposes, then reuse it widely."

      • How does domain-specific design meet the semantic challenge? "By defining a document format that represents the problem and by registering a media type for that format, so that a client knows right away when it's encountered an instance of the problem." (p60)

      • What are the downsides to developing a domain-specific design? Paraphrasing: you won't leverage work by predecessors.

      • What is a link relation? "A link relation is a magical string associated with a hypermedia control like Maze+XML's <link> tag. It explains the change in application state (for safe requests) or resource state (for unsafe requests) that will happen if the client triggers the control." See also RFC 5988, and registry of link relations managed by IANA.

      • Is Maze+XML an API? (p67) It doesn't define API calls, or give any rules for constructing URLs. Still an API, or no? "My experience shows that the 'API call' metaphor inevitably exposes the server's implementation details to clients. ... When you allow someone outside your organization to make API calls, you make that person a silent partner in the implementation of your server. It becomes very difficult to change anything on the server side without hurting this unknown customer."

      • How does hypermedia give an API more flexibility? "Every time a client makes an HTTP request, the server sends a response explaining which HTTP requests make the most sense as a next step. If the server-side options change, that document changes along with it." Consider being given a sheet of paper with a list of instructions, versus a stack of index cards with one instruction per card. Which is easier to change?

      • How does a designer of a domain-specific API bridge the semantic gap?

        • Write down application semantics in a human-readable specification (like Maze+XML).

        • Register one or more IANA media types for your design (involves associating the type to your document).

      • How does a client of a domain-specific API cross the semantic gap from their side?

        • Look up an unknown media type in the IANA registry.

        • Read the human-readable specification.

        • http://amundsen.com/media-types/maze/

        • http://amundsen.com/examples/mazes/2d

        • http://restfulie.caelum.com.br/

  • April 22nd: Chapter 6

    • Chapter 6, The Collection Pattern One pattern in particular—the collection pattern—shows up over and over again in API design. In this chapter, I show off two different standards that capture this pattern: Collection+JSON and AtomPub

    • Discussion

      • Fun fact, I have issues open on 2 out of 3 of these gems:

        • https://github.com/carlesjove/collection_json_rails

        • https://github.com/apotonick/roar-rails

        • https://github.com/rails-api/active_model_serializers

      • Why use a collection pattern standard? You'll be able to focus on adapting your application semantics to that pattern, and you'll get access to a preexisting base of client programs and server-side tools. (p92)

      • What is a collection resource? It exists mainly to group other resources together. Its representation focuses on links to other resources, though it may include snippets from the representations.

      • What are individual resources contained within a collection called? Item, entry, or member. Things listed in a collection have independent existence.

      • What are the major components of Colletion+JSON? href, items, queries, and template (write template). An item has a permanent link, and its own data.

      • Does Collection+JSON have explicit support for pagination? No. :c

      • Why didn't AtomPub catch on? (According to the book.) XML.

      • the semantic challenge is: How can we program a computer to decide which links to click? 

        • where does Collection+JSON put its application semantics? in prompt tags

        • IANA registered link relations make a lot more sense in the context of a collection pattern

          • http://www.iana.org/assignments/link-relations/link-relations.xhtml

  • May 6th: Chapter 7

    • Chapter 7, Pure-Hypermedia Designs When the collection pattern doesn’t fit your requirements, you can convey any representation you want using a general-purpose hypermedia format. This chapter shows how it works using three general hypermedia formats (HTML, HAL, and Siren) as examples. This chapter also introduces HTML microformats and micro‐ data, which lead in to the next chapter. 

      • Discussion
        • vCard vs hCard
          • what was wrong with hCard? not hypermedia: which means, no link to related resources
          • hCard leveraged the time and money invested in establishing a standard, but translated it into a hypermedia format: HTML (p114)
          • what is a microformat? "Microformats let you add extra application semantics to HTML." (p113)
        • what is Microdata? a refinement of microformat.
          • how is microformat a "hack"? "HTML’s class attribute was designed to convey information about visual display (via CSS), not to convey bits of application semantics." (p116)

          • HTML Microdata. introduces five new attributes specifically for representing application semantics: itemprop, itemscope, itemtype, itemid, and itemref. These at‐ tributes may appear on any HTML tag. 

          • How supported is it? http://en.wikipedia.org/wiki/Microdata_%28HTML%29#Support

        • What is the difference between changing application state and changing resource state? Application state is what the client sees, resource state is what the server sees.

        • "you should design APIs by identifying the resources and tying them together with hypermedia." vs "in a hypermedia-based design, resources don’t matter as much. The designer’s job is to identify all the state transitions." (p123)

        • what are the limits of HTML/HTML4? (p124) HTML is technically domain-specific: human readable documents. in 4:

          • HTML lacks full http protocol support (no put or delete)

          • forms can only build entity-bodies in two different formats

          • doesn't distinguish between strings and numbers

          • doesn't define a way of representing dates!!!

        • how will HTML5 help?

          • defines TIME TAG!!!!!!!!!!

          • new hypermedia controls, including audio, video, source, embed

          • new options for validating input tags
          • microdata
        • how won't HTML5 help? :c
          • still no put or delete
          • form media type only extend to text/plain
        • so what is HAL? Hypermedia Application Language
          • focuses on hyperlink
          • what does HAL define? resources and links
          • problems with HAL? strips way too much, no request direction, basically like the overloaded-post semantics only
        • what's wrong with the API documentation HAL needs?
          • "It spends a lot of time detailing the HTTP request necessary to trigger a state transition, and only mentions in passing that the purpose of the state transition is to create a new account." (p128)
        • what are the pros and cons of SIREN?

          • "The upside of this approach is greater flexibility in representing state transitions that don’t fit the collection pattern. The downside is that two Siren applications (or two HTML applications) will have less in common, and require more special client-side programming, than two Collection+JSON applications." (p130)

        • Semantic challenge: "Given an API, how can a client developer write a computer program that makes decisions based on the API’s application semantics?"
  • May 13th: Chapter 8

    • Chapter 8, Profiles A profile fills in the gaps between a data format (which can be used by many different APIs) and a specific API implementation. The profile format I recommend is ALPS, but I also cover XMDP and JSON-LD. In this chapter, my advice begins to outstrip the state of the art at the time this book was written. I had to develop the ALPS format for this book, because nothing else would do the job. If you’re already familiar with hypermedia-based designs, you might be able to skip up to Chapter 8, but I don’t think you should skip past it.

      • Discussion

        • API design steps (p 133)

          •  

            • Is there a domain-specific standard for your problem? If so, use it. Document any application-specific extensions (Chapter 5).

            • Does your problem fit the collection pattern? If so, adopt one of the collection standards. Define an application-specific vocabulary and document it (Chapter 6).

            • If neither of those is true, choose a general hypermedia format. Break down your application into its state transitions. Document those state transitions (Chapter 7).

            • At this point, you have your protocol semantics nailed down. The application se‐ mantics are all that remain. Are there existing microdata items or microformats that cover your problem domain? If so, use them. Otherwise, define an application- specific vocabulary and document it (Chapter 7). 

        • How do you choose a hypermedia format that best allows you to express the state transitions of your API?

        • How should clients find documentation?

          • Fielding constraint is "self-describing messages." HTTP's Content-Type header is an example. If the media type defines hypermedia controls, then parsing the response document tells you which HTTP requests you can make next. Boom, those are the protocol semantics. If the media type is domain specific, it will also give you the application semantics.

          • What about media types that don't give you application semantics (say, html with hCard) or even protocol semantics (json)? You need a "profile."

          • What is a profile? "A profile is defined to not alter the semantics of the resource representation itself, but to allow clients to learn about additional semantics... associated with the resource repre‐ sentation, in addition to those defined by the media type... " (p135)

            • hCard example. HTML is a domain specific media type for "human readable document", but what if we want to process all of a set of documents as something with additional meaning, say "people"? By parsing the hCard semantics, we can do that. How does a computer know a document represents a "person" vs a "car"? Without some ontology, spec, profile, etc., it can't!

          • Finally we can answer: clients should find documentation via links! Specifically, we can use RFC 6906, which defines a link relation called profile.  
        • Where/how can I specify link relations for profiles? (p136)
        • When does a profile need to specify protocol semantics? (Vs. just application semantics.)

          • A profile only needs to describe protocol semantics when the media type has no hypermedia controls (as with JSON), or when the controls aren’t specific enough to explain exactly which HTTP request the client should make (as with HAL). 

        • What is the difference between a link relation and a semantic descriptor? "a link relation is a magical string attached to a hypermedia control, which describes the state transition that will occur if the client triggers the control." If you click "profile" link, you will move application state to that link, see? These can be safe or unsafe. A semantic descriptor is a signal about application semantics.
          •  

            • A link relation is a string describing the state transition that will happen if the client triggers a hypermedia control. Example: Maze+XML’s east relation, which lets you know that a certain link points to something geographically east of the current resource. Traditionally, the state transition is a change in application state (triggered with a GET request), but it can also be a change in resource state (triggered by PUT, POST, DELETE, or PATCH).

            • A semantic descriptor is a short string that indicates what some part of a represen‐ tation means. Example: hCard’s fn descriptor, which is used as a CSS class to mark up a person’s name in HTML. Unlike “link relation,” this is a term I made up for this book. 

        • What did XMDP and what does ALPS try to do? Machine readable profiles. What did XMDP lack that ALPS achieves? XMDP is HTML-specific. What if your representation format is something else? JSON, XML? You'd want a profile that is agnostic about representation format: one that would work on HTML, JSON, anything!
        • What does JSON-LD do? Kinda like XMDP, except for JSON. Adds a "context" to a json document, to give it more semantics.
  • May 20th: Chapter 9

    • Chapter 9, The Design Procedure This chapter brings together everything discussed in the book so far, and gives a step-by-step guide to designing a RESTful API.

      • Discussion

        • What is the simplest form of the procedure suggested by this book? 

          1. Choose a media type to use in your representations. 

          2. Write a profile that covers the rest. (p157)

        • We're used to spending all our time in step 2, but what if we spend a little more time upfront finding pieces to work in 1?

        • What's the seven step design procedure?

          1. Semantic descriptors: list all info a client might have as input/output

          2. Link relations: Draw a state diagram

            What do we understand at this point in the process? Protocol semantics.

          3. Check against existing profiles: can you leverage any?

          4. Choose a media type

          5. Write your profile

            The more you found in 3 and 4, the less you need to do in 5.

          6. Code!!

          7. Publish the "billboard URL"

        • These are listed in the book as: "List the semantic descriptors, Draw a state diagram, Reconcile names, Choose a media type, Write a profile, Implementation, Publication" (p159-170)
        • What kind of general advice is given about design?
          • Resources are implementation details
          • Don't use your database schema as the basis for your API design
          • Don't start with a representation format
          • URL design is overblown (he says it doesn't matter)
          • Standard names are better than your names

  • June 10th: Chapter 11

    • Chapter 11, HTTP for APIs This chapter gives some best practices for the use of HTTP in API implementations. I also discuss some extensions to HTTP, including the forthcoming HTTP 2.0 pro‐ tocol. 

      • Discussion

        • Take a step back from thinking about the web as merely a transport protocol, instead think of it as its own application. What is the web's "tech stack"? URLs, HTTP protocol, hypermedia on top. They answer the questions, in order, "where is the resource?" "how do I communicate with the resource?" "what next?"

        • What stuff does HTTP 1.1 give us? Response codes, headers, content negotiation.

        • How do we mark a canonical URL? Hypermedia controls: Content-Location header, or IANA-registered link relation canonical.

        • Cache controls: max-age, no-cache, Last-Modified, If-Modified-Since, ETag

        • Why do we need to be careful about keeping cached resources normalized? "If you have a collection resource whose representation includes bits of other representations, that resource’s Last-Modified represents the last time any of that stuff changed." Thus if you glom together a bunch of resources, there's a much higher chance of one tiny piece changing, but then you'll be sending the entire glommed document together over the wire.

        • What is a "Look Before You Leap" request? Pairing the Expect: 100-continue header with status code 100/417 (Expectation Failed). It saves us sending huge representations to the server, when the server won't accept them. Consider huge file uploads.

        • In a partial GET, can Accept-Ranges take values other than "bytes"?

        • What can we use If-None-Match and If-Unmodified-Since for? Data loss due to race conditions on PUTs and PATCHes. What do statuses 412 and 428 mean? Precondition failed, precondition required.

        • What's wrong with Basic Auth? Unsecure without SSL. What's wrong with it and Digest? If one client is untrusted, all need to update their passwords.

        • OAuth: client based credentials. What did 1 lack that 2 provided? Expands the processes to get an access token. When to give up on OAuth? Don't give up on OAuth.

        • HTTP Extensions

          • PATCH

          • What do the LINK and UNLINK do? Or, How can one HTTP request refer to two different resources?

            • "The LINK or UNLINK request is sent to the URL of resource A, and resource B is mentioned in the Link header. The link relation associated with the Link header describes the desired relationship between A and B." p258

          • What is WebDAV for? Protocol semantics for filesystem operations.

        • HTTP 2.0

          • What are the goals? Performance improvement without protocol semantics change. "every HTTP 1.1 request has a large setup cost. HTTP 2.0 will eliminate that setup cost."

            • Compressing headers will be legal.

            • Streams (client will be able to send multiple simultaneous requests) to a server over a single TCP connection. Pipelining in HTTP 1.1 doesn't work well – HTTP 2.0 should fix it.

          • How will this affect APIs? Special batch designs will become obsolete.
  • July 15th: Chapter 12

    • Chapter 12, Resource Description and Linked Data Linked Data is the Semantic Web community’s approach to REST. JSON-LD is arguably the most important Linked Data standard. It’s covered briefly in Chaper 8, and I revisit it here. This chapter also covers the RDF data model, and some RDF-based hypermedia formats that I didn’t get to in Chapter 10. 

      • Discussion

        • What is a representation strategy? "a client sends a GET request to the URL of a resource and receives a representation of that very resource."

        • What is a description strategy? "a representation of resource A may also have something to say about resource B"

        • Does a "pure description strategy" follow the Fielding constraints? No. They frequently describe resources that "don't exist".

        • Are URLs guaranteed to be de-referencable in RDF? No, "RDF doesn’t treat URLs as URLs. It treats them as URIs"

        • "But as far as the RDF standard is concerned, there’s no “application” to be “controlled.” Links do nothing but name the abstract connections between equally abstract resources. You’re supposed to reason about those connections, not follow them to see what’s on the other side." p266

        • Why use the resource description strategy? (Instead of the representation strategy.) 

          • "resource description lets you talk about a resource when you don’t control the representation."

          • add an "exoskeleton" to introduce hypermedia

          • "to talk about a resource that has no representation because it’s not on the Web"

        • For resources with URIs, RDF can be useful. For anonymous resources (resources with no URI) we probably want to go to representation.

        • What are resource types? (In the Semantic web, also called abstract semantic types or semantic types.) "They're classifications, like genres of books or species of animals." 
        • What is the difference between a resource type and a link relation? "A link relation and a resource type are two different things. A resource type (http:// schema.org/Person) is a statement about the resource on the other end of a link. A link relation (author or http://alps.io/iana/relations#author) is a statement about the link itself—about the relationship between the two resources."
        •  What do domain and range in RDF schema? If you think of something like birthDate as a function, you put in a Person and get out a Date.
        • Why can't you build a RESTful API on top of RDF alone? "Because RDF uses URIs instead of URLS". Ok, so what? "There’s no guarantee a client can get a representation of any of the resources it sees described. This renders most of the Fielding constraints irrelevant."
        • What are Tim Berners-Lee's 4 principles of Linked Data?
          1. Use URIs as names for things.
          2. Use HTTP URIs so that people can look up those names. (Use URLs.)
          3. When someone looks up a URI, provide useful information, using the standards (RDF*,SPARQL).
          4. Include links to other URIs so that they can discover more things.

    • Chapter 13, CoAP: REST for Embedded Systems This chapter closes out the core body of the book by covering CoAP, a RESTful protocol that doesn’t use HTTP at all.

      • Discussion

        • "a CoAP message shouldn’t be larger than about 1,024 bytes. On a low-power wireless network, you probably don’t want to go above 80 bytes." compare this to HTTP headers (not even full messages) "Uncompressed request and response headers. Request headers today vary in size from ~200 bytes to over 2KB. As applications use more cookies and user agents expand features, typical header sizes of 700-800 bytes is common." http://dev.chromium.org/spdy/spdy-whitepaper

        • What does CON mean? What about ACK? NON? RST? (p289)

        • What can CoAP do that HTTP can't? Delayed response and multicast messages. 
          • What is multicast messaging? "broadcast a message to every machine on the local network" Why can't HTTP do it? TCP doesn't support it, CoAP uses UDP.
          • What is delayed response? "a way for the server to get back in touch with the client once it’s finished processing the message that was 'accepted.'"
        • Is CoAP more or less stateless than HTTP? Less stateless "since a request and its responses are not tied together by a TCP connection"
